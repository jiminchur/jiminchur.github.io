### 1. 트랜잭션 시스템의 기본 개념

- **트랜잭션의 정의 및 특성**
    - **트랜잭션(Transaction):** 데이터베이스에서 하나의 논리적 작업 단위로, 여러 데이터 연산(read, write 등)이 하나로 묶여 수행됩니다.
    - **ACID 특성:**
    • **원자성(Atomicity):** 모든 작업이 전부 실행되거나 전혀 실행되지 않아야 함
    • **일관성(Consistency):** 트랜잭션 완료 후 데이터베이스는 항상 일관된 상태를 유지
    • **고립성(Isolation):** 각 트랜잭션은 서로 간섭 없이 독립적으로 실행됨
    • **지속성(Durability):** 커밋된 결과는 영구적으로 반영됨
- **시스템 유형**
    - **단일 사용자 시스템:** 한 번에 한 명만 사용
    - **다수 사용자 시스템:** 여러 사용자가 동시에 접근하며, 병행 처리가 필수적임


### 2. 동시성 제어와 직렬 스케줄

- **스케줄(Schedule) 개념**
    - 여러 트랜잭션의 연산들이 시간 순서대로 배열된 실행 기록이며, 각 트랜잭션 내에서는 원래 순서가 유지되어야 합니다.
- **직렬 스케줄 vs. 비직렬 스케줄**
    - **직렬 스케줄:** 트랜잭션들이 완전히 분리되어 순차적으로 실행되어, 한 트랜잭션의 연산이 다른 트랜잭션과 섞이지 않음
    - **비직렬 스케줄:** 여러 트랜잭션의 연산들이 서로 인터리빙되어 실행되지만, 충돌 순서를 고려하면 직렬 스케줄과 동일한 결과를 낼 수 있음
- **충돌과 직렬 가능성 (Serializability)**
    - **충돌 연산:** 같은 데이터 항목에 대해 두 트랜잭션 중 적어도 한 쪽이 쓰기 연산을 수행할 때 발생
    - **선행 그래프(Precedence Graph):** 트랜잭션 간의 충돌 관계를 그래프로 표현하여, 사이클이 없으면 직렬 가능하다고 판단


### 3. SQL의 트랜잭션 지원 및 고립성 등급

- **트랜잭션 시작 및 종료**
    - SQL에서는 명시적 Begin Transaction 없이 SQL 문 실행 시 묵시적으로 트랜잭션이 시작되고, COMMIT 또는 ROLLBACK으로 종료됩니다.
- **트랜잭션 속성 설정**
    - **접근 모드:** READ ONLY 또는 READ WRITE
    - **고립성 등급:** READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE 등
    - 낮은 고립성 등급은 오손 읽기(Dirty Read), 반복 불가능 읽기(Unrepeatable Read), 팬텀 현상(Phantom) 등의 문제를 발생시킬 수 있음


### 4. 동시성 제어 기법

- **2단계 로킹 (Two-Phase Locking, 2PL)**
    - **기본 원리:** 트랜잭션은 로크를 획득하는 증분 단계(Growing Phase) 후, 해제하는 감소 단계(Shrinking Phase)를 거쳐야 하며, 이를 준수하면 직렬 가능성이 보장됩니다.
    - **공유/배타 로크:** 읽기 작업은 공유 로크, 쓰기 작업은 배타적 로크를 사용합니다.
- **타임스탬프 기반 동시성 제어**
    - 각 트랜잭션에 고유 타임스탬프를 부여하여, 데이터 항목의 읽기/쓰기 타임스탬프와 비교합니다. 순서 위반 시 해당 트랜잭션을 철회하거나 대기시키며, 엄격한 타임스탬프 알고리즘은 회복 가능성과 직렬 가능성을 동시에 보장합니다.
- **낙관적 동시성 제어 (Optimistic Concurrency Control)**
    - **세 단계 프로토콜:**
        1. **읽기 단계 (Read Phase):** 데이터를 읽어 로컬 복사본에 저장
        2. **검증 단계 (Validation Phase):** 종료 직전에 다른 트랜잭션과의 충돌 여부를 검사
        3. **쓰기 단계 (Write Phase):** 검증이 성공하면 변경 사항을 데이터베이스에 반영, 실패 시 재실행
    - 간섭이 적은 환경에서 효과적으로 작동합니다.


### 5. 회복(Recovery) 기법

- **회복의 개념 및 필요성**
    - 시스템 장애나 트랜잭션 실패 시, 데이터베이스를 최근의 일관된 상태로 복원하여 지속성(Durability)과 무결성을 보장합니다.
- **즉시 갱신 vs. 지연 갱신**
    - **즉시 갱신 (Immediate Update):** 트랜잭션 실행과 동시에 디스크에 갱신을 반영하며, 장애 발생 시 UNDO 작업이 필요합니다.
    - **지연 갱신 (Deferred Update):** 트랜잭션 커밋 시점까지 실제 갱신을 연기하여, 미완료 트랜잭션은 단순히 재실행할 수 있습니다.
- **로그 관리 및 체크포인트**
    - **WAL (Write-Ahead Logging):** 데이터 변경 전 로그를 디스크에 기록하여, 복구 시 UNDO/REDO 작업에 활용합니다.
    - **체크포인트:** 주기적으로 버퍼의 변경 데이터를 디스크에 강제로 기록하여, 복구 시 처리해야 할 트랜잭션 수를 줄입니다.

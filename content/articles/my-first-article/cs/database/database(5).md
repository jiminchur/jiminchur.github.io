## **1. ER 모델과 관계 모델 변환 개요**

- **ER 모델 (Entity-Relationship Model)**
    - 개념적 데이터 모델로, 현실 세계의 데이터를 **엔티티(Entity), 속성(Attribute), 관계(Relationship)** 등의 개념을 사용해 표현
    - 다이어그램을 이용하여 구조를 시각적으로 표현 (ER 다이어그램)
    - 설계 시 **엔티티, 관계, 속성, 제약조건** 등을 정의함
- **관계 모델 (Relational Model)**
    - 관계형 데이터베이스에서 데이터를 **테이블(Relation)** 로 표현하는 논리적 모델
    - **키(Key), 무결성 제약조건(Integrity Constraints)** 을 기반으로 데이터의 일관성을 유지
    - 데이터를 **정규화(Normalization)** 하여 중복을 줄이고 데이터 무결성을 향상

---

## **2. ER 모델 → 관계 모델 변환 (Mapping)**

ER 모델을 관계 모델로 변환하는 방법을 학습합니다.

### **(1) 일반 엔티티 변환**

- **일반적인 엔티티(Strong Entity) → 하나의 릴레이션(테이블)로 변환**
    - 각 속성은 릴레이션의 컬럼(속성)으로 매핑됨
    - 기본키(Primary Key)는 엔티티의 고유 식별자(예: 학번, 사원번호 등)
    
    **예제)** `STUDENT(학번, 이름, 학과, 학년)`
    
    → 관계 모델: **STUDENT(학번(PK), 이름, 학과, 학년)**
    

### **(2) 약한 엔티티 변환**

- **약한 엔티티(Weak Entity) → 별도 릴레이션 생성, 주 엔티티의 기본키를 포함하여 식별**
    - 기본키를 가지지 않으며, **주 엔티티의 기본키 + 부분 키** 조합을 기본키로 설정
    - 주 엔티티와 약한 엔티티 사이에 **식별 관계(Identifying Relationship)** 가 존재
    
    **예제)** `DEPENDENT(피부양자 이름, 관계, 생년월일)`
    
    → 관계 모델: **DEPENDENT(사원번호(PK, FK), 피부양자이름(PK), 관계, 생년월일)**
    

### **(3) 1:1 관계 변환**

- 한쪽에 **외래키(Foreign Key)** 를 추가하여 관계 표현
    - **전체 참여(Total Participation)** 인 경우, 하나의 릴레이션으로 합칠 수도 있음
    
    **예제)** `EMPLOYEE(사원번호, 이름, 부서번호)` - `MANAGER(사원번호, 시작일)`
    
    → 관계 모델: **EMPLOYEE(사원번호(PK), 이름, 부서번호, 시작일)**
    

### **(4) 1:N 관계 변환**

- **N 쪽(Relation) 테이블에 1쪽의 기본키를 외래키(FK)로 추가**
    - 즉, 다수(N) 관계에 있는 테이블이 1쪽의 기본키를 참조하는 **외래키(Foreign Key)** 를 가짐
    
    **예제)** `DEPARTMENT(부서번호, 부서이름)` - `EMPLOYEE(사원번호, 이름, 부서번호(FK))`
    
    → 관계 모델: **EMPLOYEE(사원번호(PK), 이름, 부서번호(FK))**
    

### **(5) N:M 관계 변환**

- 별도의 **관계(Relation) 테이블** 을 생성하여 양쪽의 기본키를 **복합키(Composite Key)** 로 설정
    - 두 테이블 간의 **교차 테이블(Join Table)** 역할을 함
    
    **예제)** `STUDENT(학번, 이름)` - `COURSE(과목번호, 과목명)` - 관계: `수강`
    
    → 관계 모델: **ENROLL(학번(PK, FK), 과목번호(PK, FK), 학점, 수강년도)**
    

### **(6) 다치 속성 변환**

- **다치 속성(Multi-valued Attribute) → 별도의 테이블로 변환**
    - 원래 엔티티의 기본키를 포함하여 다치 속성을 표현
    
    **예제)** `PROJECT(프로젝트번호, 프로젝트이름, 위치(다치 속성))`
    
    → 관계 모델: **PROJECT_LOCATION(프로젝트번호(PK, FK), 위치(PK))**

## **3. 관계형 모델의 키와 무결성 제약조건**

ER 모델을 관계형 데이터베이스에 적용할 때, **무결성 제약조건(Integrity Constraints)** 을 고려해야 합니다.

### **(1) 기본키(Primary Key)**

- 각 릴레이션에서 **투플을 유일하게 식별**하는 속성
- NULL 값이 포함될 수 없음

### **(2) 외래키(Foreign Key)**

- 다른 테이블의 기본키를 참조하는 속성
- 외래키 값은 참조된 릴레이션의 기본키 중 하나여야 함
    
    **예제)**
    
    `EMPLOYEE(사원번호(PK), 이름, 부서번호(FK))`
    
    `DEPARTMENT(부서번호(PK), 부서이름)`
    
    → **EMPLOYEE의 부서번호는 DEPARTMENT의 부서번호를 참조(Foreign Key).**
    

### **(3) 무결성 제약조건**

- **개체 무결성(Entity Integrity):** 기본키는 NULL이 될 수 없음
- **참조 무결성(Referential Integrity):** 외래키는 참조하는 기본키 값과 일치해야 함

## **4. 관계 데이터베이스 정규화 (Normalization)**

- **정규화(Normalization):** 데이터 중복을 최소화하고, 데이터 무결성을 유지하는 과정
- **정규형(Normal Form)**
    - **1NF (제1정규형):** 모든 속성이 **원자 값(Atomic Value)** 을 가져야 함
    - **2NF (제2정규형):** 기본키가 아닌 속성들이 **부분 함수적 종속(Partial Dependency)** 을 가지지 않도록 분해
    - **3NF (제3정규형):** **이행적 종속(Transitive Dependency)** 을 제거
    - **BCNF (보이스-코드 정규형):** 모든 결정자가 후보키를 만족하도록 개선
    
    **예제)**
    
    `STUDENT(학번, 이름, 학과, 과목코드, 교수)`
    
    → 2NF로 변환: `STUDENT(학번, 이름, 학과)` + `ENROLL(학번, 과목코드, 교수)`

## **5. 요약 및 학습 포인트**

- **ER 모델 → 관계 모델 변환 과정 숙지**
    - 각 엔티티와 관계를 테이블로 변환하는 방법을 이해해야 합니다.
- **N:M 관계 및 다치 속성 변환**
    - 별도의 테이블을 생성하여 외래키를 포함하는 방식으로 변환
- **키와 무결성 제약조건을 활용한 데이터베이스 설계**
    - 기본키와 외래키를 정확히 설정하여 데이터의 무결성을 유지
- **정규화를 통해 중복을 줄이고 일관성 유지**
    - 데이터 중복을 최소화하고 이상(Anomaly) 방지를 위한 정규화 단계 숙지
